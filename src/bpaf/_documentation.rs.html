29        //! #### Introduction and design goals
30        //! A quick intro. What, why and how
31        //! 
32        //! `bpaf` is a lightweight and flexible command line parser that uses both combinatoric and derive
33        //! style API
34        //! 
35        //! Combinatoric API usually means a bit more typing but no dependency on proc macros and more help
36        //! from the IDE, derive API uses proc macro to save on typing but your IDE will be less likely to
37        //! help you. Picking one API style does not lock you out from using the other style, you can mix
38        //! and match both in a single parser
39        //! 
40        //! # Examples of both styles
41        //! 
42        #![cfg_attr(not(doctest), doc = include_str!("docs2/intro.md"))]
43        //! 
44        //! # Design goals
45        //! 
46        //! ## Parse, don't validate
47        //! 
48        //! `bpaf` tries hard to let you move as many invariants about the user input you are
49        //! trying to parse into rust types: for mutually exclusive options you can get `enum` with
50        //! exclusive items going into separate branches, and you can collect results into types like
51        //! [`BTreeSet`](std::collections::BTreeSet), or whatever custom type you might have with
52        //! custom parsing. Ideas for
53        //! [making invalid states unrepresentable](https://geeklaunch.io/blog/make-invalid-states-unrepresentable/)
54        //! and [using parsing over validation](https://lexi-lambda.github.io/blog/2019/11/05/parse-don-t-validate/)
55        //! are not new.
56        //! 
57        //! That said you can also validate your inputs if this fits your situation better. If you want to
58        //! ensure that the sum of every numeric field must be divisible by both 3 and 5, but only when it's
59        //! Thursday - you can do that too.
60        //! 
61        //! ## Flexibility
62        //! 
63        //! While aiming to be a general-purpose command line parser `bpaf` offers a few backdoors that
64        //! allow you to parse pretty much anything you want: chained commands, custom blocks of options,
65        //! DOS-style options (`/ofile.pas`), `dd` style options (`if=file of=out`), etc. A similar idea applies
66        //! to what the parser can produce - if your app operates with boxed string slices internally - `bpaf`
67        //! will give you `Box<str>` instead of `String` if you ask it to.
68        //! 
69        //! The only restriction is that you cannot use information from items parsed earlier (but not
70        //! the fact that something was parsed successfully or not) to decide to how to parse further
71        //! options, and even then you can side step this restriction by passing some shared state as a
72        //! parameter to the parsers.
73        //! 
74        //! ## Reusability
75        //! 
76        //! Parsers in `bpaf` are not monolithic and you can share their parts across multiple binaries,
77        //! workspace members or even independent projects. Say you have multiple binaries in a workspace
78        //! that perform different operations on some input. You can declare a parser for the input
79        //! specifically, along with all the validations, help messages or shell dynamic completion
80        //! functions you need and use it across all the binaries alongside the arguments specific to
81        //! those binaries.
82        //! 
83        //! ## Composition, transformation
84        //! 
85        //! Parsers in `bpaf` are not finalized either, say you have a parser that describes a single input
86        //! for your program, it can take multiple arguments or perform extra validations, etc. You can
87        //! always compose this parser with any other parser to produce tuples of both results for example.
88        //! Or to make it so parser runs multiple times and collects results into a `Vec`.
89        //! 
90        //! ## Performance
91        //! 
92        //! While performance is an explicit non-goal - `bpaf` does nothing that would pessimize it either,
93        //! so performance is on par or better compared to other fully featured parsers.
94        //! 
95        //! ## Correctness
96        //! 
97        //! `bpaf` would parse only items it can represent and will reject anything it cannot represent
98        //! in the output. Say your parser accepts both `--intel` and `--att` flags, but encodes the result
99        //! into `enum Style { Intel, Att }`, `bpaf` will accept those flags separately, but not if they
100        //! are used both at once. If the parser later collects multiple styles into a `Vec<Style>` then it
101        //! will accept any combinationof those flags.
102        //! 
103        //! ## User friendly
104        //! 
105        //! `bpaf` tries to provide user-friendly error messages, and suggestions for typos but also scripts
106        //! for shell completion, `man` pages and markdown documentation for the web.
107        //!

589            //! #### Combinatoric API
590            //! Parse arguments without using proc macros
591            //! 
592            //! When making a parser in the Combinatoric style API you usually go through those steps
593            //! 
594            //! 1. Design data type your application will receive
595            //! 2. Design command line options user will have to pass
596            //! 3. Create a set of simple parsers
597            //! 4. Combine and transform simple parsers to create the final data type
598            //! 5. Transform the resulting [`Parser`] into [`OptionParser`] and run it
599            //! 
600            //! Let's go through some of them in more detail:

689                    //! #### Switch parser
690                    //! 
691                    //! Let's start with the simplest possible one - a simple switch that gets parsed into a `bool`.
692                    //! 
693                    //! First of all - the switch needs a name - you can start with [`short`] or [`long`] and add more
694                    //! names if you want: `long("simple")` or `short('s').long("simple")`. This gives something with
695                    //! the type [`NamedArg`]:
696                    //! 
697                    //! ```rust
698                    //! # use bpaf::*;
699                    //! use bpaf::parsers::NamedArg;
700                    //! fn simple_switch() -> NamedArg {
701                    //!     short('s').long("simple")
702                    //! }
703                    //! ```
704                    //! 
705                    //! From `NamedArg` you make a switch parser by calling [`NamedArg::switch`]. Usually, you do it
706                    //! right away without assigning `NamedArg` to a variable.
707                    //! 
708                    //! ```rust
709                    //! # use bpaf::*;
710                    //! fn simple_switch() -> impl Parser<bool> {
711                    //!     short('s').long("simple").switch()
712                    //! }
713                    //! ```
714                    //! 
715                    //! The switch parser we just made implements trait [`Parser`] and to run it you convert it to [`OptionParser`] with
716                    //! [`Parser::to_options`] and run it with [`OptionParser::run`]

894                //! #### Transforming parsers
895                //! 
896                //! Once you have your primitive parsers done you might want to improve them a bit - add fallback
897                //! values, or change them to consume multiple items, etc. Every primitive (or composite) parser
898                //! implements [`Parser`] so most of the transformations are coming from this trait.
899                //! 
900                //! Say you have a parser that takes a crate name as a required argument you want to use in your own
901                //! `cargo test` replacement
902                //! 
903                //! ```rust
904                //! use bpaf::*;
905                //! fn krate() -> impl Parser<String> {
906                //!     long("crate").help("Crate name to process").argument("CRATE")
907                //! }
908                //! ```
909                //! 
910                //! You can turn it into, for example, an optional argument - something that returns
911                //! `Some("my_crate")` if specified or `None` if it wasn't. Or to let the user to pass a multiple
912                //! of them and collect them all into a `Vec`
913                //! 
914                //! 
915                //! ```rust
916                //! use bpaf::*;
917                //! fn maybe_krate() -> impl Parser<Option<String>> {
918                //!     long("crate")
919                //!         .help("Crate name to process")
920                //!         .argument("CRATE")
921                //!         .optional()
922                //! }
923                //! 
924                //! fn krates() -> impl Parser<Vec<String>> {
925                //!     long("crate")
926                //!         .help("Crate name to process")
927                //!         .argument("CRATE")
928                //!         .many()
929                //! }
930                //! ```
931                //! 
932                //! A complete example:
933                #![cfg_attr(not(doctest), doc = include_str!("docs2/compose_basic_many.md"))]
934                //! 
935                //! Transforming a parser with a method from the `Parser` trait usually gives you a new parser back and
936                //! you can chain as many transformations as you need.
937                //! 
938                //! Transformations available in the `Parser` trait things like adding fallback values, making
939                //! the parser optional, making it so it consumes many but at least one value, changing how it is
940                //! being shown in `--help` output, adding additional validation and parsing on top and so on.
941                //! 
942                //! The order of those chained transformations matters and for some operations using the right order
943                //! makes code cleaner. For example, suppose you are trying to write a parser that takes an even
944                //! number and this parser should be optional. There are two ways to write it:
945                //! 
946                //! Validation first:
947                //! 
948                //! ```rust
949                //! # use bpaf::*;
950                //! fn even() -> impl Parser<Option<usize>> {
951                //!     long("even")
952                //!         .argument("N")
953                //!         .guard(|&n| n % 2 == 0, "number must be even")
954                //!         .optional()
955                //! }
956                //! ```
957                //! 
958                //! Optional first:
959                //! 
960                //! ```rust
961                //! # use bpaf::*;
962                //! fn even() -> impl Parser<Option<usize>> {
963                //!     long("even")
964                //!         .argument("N")
965                //!         .optional()
966                //!         .guard(|&n| n.map_or(true, |n| n % 2 == 0), "number must be even")
967                //! }
968                //! ```
969                //! 
970                //! In later case validation function must deal with a possibility where a number is absent, for this
971                //! specific example it makes code less readable.
972                //! 
973                //! One of the important types of transformations you can apply is a set of failing
974                //! transformations. Suppose your application operates with numbers and uses `newtype` pattern to
975                //! keep track of what numbers are odd or even. A parser that consumes an even number can use
976                //! [`Parser::parse`] and may look like this:
977                //! 
978                //! ```rust
979                //! # use bpaf::*;
980                //! pub struct Even(usize);
981                //! 
982                //! fn mk_even(n: usize) -> Result<Even, &'static str> {
983                //!     if n % 2 == 0 {
984                //!         Ok(Even(n))
985                //!     } else {
986                //!         Err("Not an even number")
987                //!     }
988                //! }
989                //! 
990                //! fn even() -> impl Parser<Even> {
991                //!     long("even")
992                //!         .argument::<usize>("N")
993                //!         .parse(mk_even)
994                //! }
995                //! ```
996                //!

1041                //! #### Combining multiple simple parsers
1042                //! 
1043                //! A single-item option parser can only get you so far. Fortunately, you can combine multiple
1044                //! parsers with [`construct!`] macro.
1045                //! 
1046                //! For sequential composition (all the fields must be present) you write your code as if you are
1047                //! constructing a structure, enum variant or a tuple and wrap it with `construct!`. Both
1048                //! a constructor and parsers must be present in the scope. If instead of a parser you have a function
1049                //! that creates one - just add `()` after the name:
1050                //! 
1051                //! ```rust
1052                //! # use bpaf::*;
1053                //! struct Options {
1054                //!     alpha: usize,
1055                //!     beta: usize
1056                //! }
1057                //! 
1058                //! fn alpha() -> impl Parser<usize> {
1059                //!     long("alpha").argument("ALPHA")
1060                //! }
1061                //! 
1062                //! fn both() -> impl Parser<Options> {
1063                //!     let beta = long("beta").argument("BETA");
1064                //!     // call `alpha` function, and use result to make parser
1065                //!     // for field `alpha`, use parser `beta` for field `beta`
1066                //!     construct!(Options { alpha(), beta })
1067                //! }
1068                //! ```
1069                //! 
1070                //! Full example:
1071                #![cfg_attr(not(doctest), doc = include_str!("docs2/compose_basic_construct.md"))]
1072                //! 
1073                //! If you are using positional parsers - they must go to the right-most side and will run in
1074                //! the order you specify them. For named parsers order affects only the `--help` message.
1075                //! 
1076                //! The second type of composition `construct!` offers is a parallel composition. You pass multiple
1077                //! parsers that produce the same result type in `[]` and `bpaf` selects one that fits best with
1078                //! the data user gave.
1079                //! 
1080                //! 
1081                #![cfg_attr(not(doctest), doc = include_str!("docs2/compose_basic_choice.md"))]
1082                //! 
1083                //! If parsers inside parallel composition can parse the same object - the longest possible match
1084                //! should go first since `bpaf` picks an earlier parser if everything else is equal, otherwise it
1085                //! does not matter. In this example `construct!([miles, km])` produces the same results as
1086                //! `construct!([km, miles])` and only `--help` message is going to be different.
1087                //! 
1088                //! Parsers created with [`construct!`] still implement the [`Parser`] trait so you can apply more
1089                //! transformation on top. For example same as you can make a simple parser optional - you can make
1090                //! a composite parser optional. Parser transformed this way will succeed if both `--alpha` and
1091                //! `--beta` are present or neither of them:
1092                //! 
1093                //! ```rust
1094                //! # use bpaf::*;
1095                //! struct Options {
1096                //!     alpha: usize,
1097                //!     beta: usize
1098                //! }
1099                //! 
1100                //! fn parser() -> impl Parser<Option<Options>> {
1101                //!     let alpha = long("alpha").argument("ALPHA");
1102                //!     let beta = long("beta").argument("BETA");
1103                //!     construct!(Options { alpha, beta }).optional()
1104                //! }
1105                //! ```

1892            //! #### Designing a good datatype
1893            //! bpaf allows you to reduce the size of legal values to valid ones
1894            //! 
1895            //! Parsing usually starts with deciding what kind of data your application wants to get from the user.
1896            //! You should try to take advantage of the Rust type system, try to represent the result such that more
1897            //! validation can be done during parsing.
1898            //! 
1899            //! Data types can represent a set of *legal* states - for example, for u8 this is all the numbers
1900            //! from 0 to 255, while your app logic may only operate correctly only on some set of *valid*
1901            //! states: if this u8 represents a fill ratio for something in percents - only valid numbers are
1902            //! from 0 to 100. You can try to narrow down the set of legal states to valid states with [newtype
1903            //! pattern](https://doc.rust-lang.org/rust-by-example/generics/new_types.html). This newtype will
1904            //! indicate through the type when you've already done validation. For the fill ratio example you can
1905            //! implement a newtype along with `FromStr` implementation to get validation for free during
1906            //! parsing.
1907            //! 
1908            //! 
1909            //! ```no_run
1910            //! # use std::str::FromStr;
1911            //! # use bpaf::*;
1912            //! #[derive(Debug, Clone, Copy)]
1913            //! pub struct Ratio(u8);
1914            //! 
1915            //! impl FromStr for Ratio {
1916            //!     type Err = &'static str;
1917            //! 
1918            //!     fn from_str(s: &str) -> Result<Self, Self::Err> {
1919            //!         match s.parse() {
1920            //!             Ok(n) if n <= 100 => Ok(Ratio(n)),
1921            //!             _ => Err("Invalid fill ratio")
1922            //!         }
1923            //!     }
1924            //! }
1925            //! 
1926            //! #[derive(Debug, Clone, Bpaf)]
1927            //! #[bpaf(options)]
1928            //! struct Options {
1929            //!     /// Fill ratio
1930            //!     ratio: Ratio
1931            //! }
1932            //! 
1933            //! fn main() {
1934            //!     println!("{:?}", options().run());
1935            //! }
1936            //! ```
1937            //! 
1938            //! 
1939            //! Try using enums instead of structs for mutually exclusive options:
1940            //! 
1941            //! ```no_check
1942            //! /// Good format selection
1943            //! #[derive(Debug, Clone, Bpaf)]
1944            //! #[bpaf(options)]
1945            //! enum OutputFormat {
1946            //!     Intel,
1947            //!     Att,
1948            //!     Llvm
1949            //! }
1950            //! 
1951            //! fn main() {
1952            //!     let format = output_format().run();
1953            //! 
1954            //!     // `rustc` ensures you handle each case, parser won't try to consume
1955            //!     // combinations of flags it can't represent. For example it won't accept
1956            //!     // both `--intel` and `--att` at once
1957            //!     // (unless it can collect multiple of them in a vector)
1958            //!     match format {
1959            //!         OutputFormat::Intel => ...,
1960            //!         OutputFormat::Att => ...,
1961            //!         OutputFormat::Llvm => ...,
1962            //!     }
1963            //! }
1964            //! ```
1965            //! 
1966            //! While it's easy to see how flags like `--intel` and `--att` maps to each of those bools,
1967            //! consuming inside your app is more fragile
1968            //! 
1969            //! ```no_check
1970            //! /// Bad format selection
1971            //! #[derive(Debug, Clone, Bpaf)]
1972            //! #[bpaf(options)]
1973            //! struct OutputFormat {
1974            //!     intel: bool,
1975            //!     att: bool,
1976            //!     llvm: bool,
1977            //! }
1978            //! 
1979            //! fn main() {
1980            //!     let format = output_format().run();
1981            //!     // what happens when none matches? Or all of them?
1982            //!     // What happens when you add a new output format?
1983            //!     if format.intel {
1984            //!         ...
1985            //!     } else if format.att {
1986            //!         ...
1987            //!     } else if format.llvm {
1988            //!         ...
1989            //!     } else {
1990            //!         // can this branch be reached?
1991            //!     }
1992            //! }
1993            //! ```
1994            //! 
1995            //! Mutually exclusive things are not limited to just flags. For example if your program can take
1996            //! input from several different sources such as file, database or interactive input it's a good
1997            //! idea to use enum as well:
1998            //! 
1999            //! ```no_check
2000            //! /// Good input selection
2001            //! #[derive(Debug, Clone, Bpaf)]
2002            //! enum Input {
2003            //!     File {
2004            //!         filepath: PathBuf,
2005            //!     }
2006            //!     Database {
2007            //!         user: String,
2008            //!         password: String.
2009            //!     }
2010            //!     Interactive,
2011            //! }
2012            //! ```
2013            //! 
2014            //! If your codebase uses newtype pattern - it's a good idea to use it starting from the command
2015            //! options:
2016            //! 
2017            //! ```no_check
2018            //! #[derive(Debug, Clone, Bpaf)]
2019            //! struct Options {
2020            //!     // better than taking a String and parsing internally
2021            //!     date: NaiveDate,
2022            //!     // f64 might work too, but you can start from some basic sanity checks

2024            //! }

2354            //! #### `find(1)`: `find -exec commands -flags terminated by \;`
2355            //! 
2356            //! An Example for `find` shows how to implement 3 different unusual options:
2357            //! 
2358            //! - an option with a long name but a single dash as a prefix: `-user bob`
2359            //! - an option that captures everything until the next fixed character
2360            //! - an option that takes a set of characters: `-mode -rw`, `mode /rw`
2361            //! 
2362            //! In all cases, long name with a single dash is implemented by the [`literal`] with
2363            //! [`ParseAny::anywhere`](crate::parsers::ParseAny::anywhere) with some items made `adjacent` to it.
2364            //! 
2365            //! To parse `-user bob` this is simply literal `-user` adjacent to a positional item with `map` to
2366            //! focus on the interesting part.
2367            //! 
2368            //! For `-exec more things here ;` this is a combination of literal `-exec`, followed by `many`
2369            //! items that are not `;` parsed positionally with `any` followed by `;` - again with `any`, but
2370            //! `literal` works too.
2371            //! 
2372            //! And lastly to parse mode - after the tag, we accept `any` to be able to handle a combination of
2373            //! modes that may or may not start with `-` and use [`Parser::parse`] to parse them or fail.
2374            //! 
2375            //! All special cases are made optional with [`Parser::optional`], but [`Parser::fallback`] also
2376            //! works.
2377            //! 
2378            #![cfg_attr(not(doctest), doc = include_str!("docs2/find.md"))]

2944        //! Alternative to create a composable interface. Exposed API and the fact that individual
2945        //! components obey certain laws ensures that any composition of parsers is valid even if it
2946        //! doesn't make any sense.
2947        //! 
2948        //! ## Category theory
2949        //! 
2950        //! Category theory, also called Abstract Nonsense, is a general theory about mathematical
2951        //! structures and their relations. *Category* in CT constists of two sorts of abstractions:
2952        //! *objects* and *morphisms* along with some extra rules:
2953        //! - objects don't expose any information other than the name and only serve as start and end points for morphisms
2954        //! - morphisms must compose with associative composition
2955        //! - there must be an *identity morphism* for every object that maps the object to itself
2956        //! 
2957        //! A simple example of a category would be a category where objects are Rust types (here: `u8` ..
2958        //! `u64`) and morphisms are functions between those types (here: `a`, `b` and `c`):
2959        //! 
2960        //! ```rust
2961        //! fn a(i: u8) -> u16 {
2962        //!     3000 + i as u16
2963        //! }
2964        //! 
2965        //! fn b(i: u16) -> u32 {
2966        //!     40000 + i as u32
2967        //! }
2968        //! 
2969        //! fn c(i: u32) -> u64 {
2970        //!     40000 + i as u64
2971        //! }
2972        //! 
2973        //! /// Identity morphism
2974        //! fn id<T>(i: T) -> T {
2975        //!     i
2976        //! }
2977        //! 
2978        //! /// morphism composition:
2979        //! /// `comp (a, comp(b, c))` gives the same results as `comp(comp(a, b), c)`
2980        //! fn comp<F, G, A, B, C>(f: F, g: G) -> impl Fn(A) -> C
2981        //! where
2982        //!     F: Fn(A) -> B,
2983        //!     G: Fn(B) -> C,
2984        //! {
2985        //!     move |i| g(f(i))
2986        //! }
2987        //! ```
2988        //! 
2989        //! ## Composition and decomposition
2990        //! 
2991        //! Decomposition is one of the keys to solving big problems - you break down big problem into a
2992        //! bunch of small problems, solve them separately and compose back a solution. Decomposition is
2993        //! not required by computers but makes it easier to think about a problem: magical number for
2994        //! human short term memory is 7 plus minus 2 objects. Category theory, studies relations and
2995        //! composition can be a valuable tool: after all decomposition only makes sense when you can
2996        //! combine components back into a solution. Imperative algorithms that operate in terms of
2997        //! mutating variables are harder decompose - individual pieces need to be aware of the variables,
2998        //! functional and declarative approaches make it easier: calculating a sum of all the numbers in a
2999        //! vector can be decomposed into running an iterator over it and applying `fold` to it: `fold`
3000        //! doesn't need to know about iteration shape, iterator doesn't need to know about how values are
3001        //! used.
3002        //! 
3003        //! In category theory you are not allowed to look inside the objects at all and can distinguish
3004        //! between them only by means of the composition so as long as implemented API obeys the
3005        //! restrictions set by category theory - it should be very composable.
3006        //! 
3007        //! ## Functors
3008        //! 
3009        //! Let's start by talking about what a `Functor` is. Wikipedia defines it as a "design pattern
3010        //! that allows for a generic type to apply a function inside without changing the structure of
3011        //! the generic type". Sounds scary, but in Rust terms it's a trait that takes a value or values
3012        //! in a container (or more general *value in a context* ) such as `Option<A>` and a function
3013        //! `fn(A) -> B` and gives you `Option<B>` back.
3014        //! 
3015        //! Closest analogy in a real code you can write in Rust right now would be modifying an `Option`
3016        //! using only `Option::map`:
3017        //! ```rust
3018        //! fn plus_one(input: Option<u32>) -> Option<u32> {
3019        //!     input.map(|i| i + 1)
3020        //! }
3021        //! 
3022        //! let present = Some(10);
3023        //! let absent = None;
3024        //! 
3025        //! assert_eq!(plus_one(present), Some(11));
3026        //! assert_eq!(plus_one(absent), None);
3027        //! ```
3028        //! 
3029        //! `Vec`, `Result` and other types that implement `map` are `Functors` as well, but `Functor`
3030        //! is not limited just to containers - you don't have to have a value inside to be able to
3031        //! manipulate it. In fact a regular rust function is also a `Functor` if you squint hard enough.
3032        //! Consider `Reader` that allows you to perform transformations on a *value in a context* `T`
3033        //! without having any value until it the execution time:
3034        //! 
3035        //! ```rust
3036        //! struct Reader<T>(Box<dyn Fn(T) -> T>);
3037        //! impl<T: 'static> Reader<T> {
3038        //!     /// Initialize an new value in a context
3039        //!     fn new() -> Self {
3040        //!         Self(Box::new(|x| x))
3041        //!     }
3042        //! 
3043        //!     /// Modify a value in a context
3044        //!     fn map<F:  Fn(T) -> T + 'static>(self, f: F) -> Self {
3045        //!         Self(Box::new(move |x| f((self.0)(x))))
3046        //!     }
3047        //! 
3048        //!     /// Apply the changes by giving it the initial value
3049        //!     fn run(self, input: T) -> T {
3050        //!         (self.0)(input)
3051        //!     }
3052        //! }
3053        //! 
3054        //! let val = Reader::<u32>::new();
3055        //! let val = val.map(|x| x + 1);
3056        //! let res = val.run(10);
3057        //! assert_eq!(res, 11);
3058        //! ```
3059        //! 
3060        //! Not all the collections are `Functors` - by `Functor` laws mapping the *value in context*
3061        //! shouldn't change the shape so any collections where shape depends on a value, such as `HashSet`
3062        //! or `BTreeSet` are out.
3063        //! 
3064        //! ## Applicative Functors
3065        //! 
3066        //! `map` in `Functor` is limited to a single *value in a context*, `Applicative Functor` extends it
3067        //! to operations combining multiple values, closest Rust analogy would be doing computations on
3068        //! `Option` or `Result` using only `?`, having `Some`/`Ok` around the whole expression and not using `return`.
3069        //! ```rust
3070        //! fn add_numbers(input_a: Option<u32>, input_b: Option<u32>) -> Option<u32> {
3071        //!     Some(input_a? + input_b?)
3072        //! }
3073        //! 
3074        //! let present_1 = Some(10);
3075        //! let present_2 = Some(20);
3076        //! let absent = None;
3077        //! 
3078        //! assert_eq!(add_numbers(present_1, present_2), Some(30));
3079        //! assert_eq!(add_numbers(present_1, absent), None);
3080        //! assert_eq!(add_numbers(absent, absent), None);
3081        //! ```
3082        //! 
3083        //! Similarly to `Functors`, `Applicative Functors` are not limited to containers and can
3084        //! represent *a value in an arbitrary context*.
3085        //! 
3086        //! `Try` trait (`?`) for `Option` and `Result` short circuits when it finds a missing value,
3087        //! but `Applicative Functors` in general don't have to - in fact to implement dynamic completion
3088        //! `bpaf` needs to check items past the first failure point to collect all the possible
3089        //! completions.
3090        //! 
3091        //! ## Alternative Functors
3092        //! 
3093        //! So far `Applicative Functors` allow us to create structs containing multiple fields out of
3094        //! individual parsers for each field. `Alternative` extends `Applicative` with two extra
3095        //! things: one for combining two *values in a context* into one and and an idenity element
3096        //! for this operation. In Rust a closest analogy would be `Option::or` and `Option::None`:
3097        //! 
3098        //! ```rust
3099        //! fn pick_number(a: Option<u32>, b: Option<u32>) -> Option<u32> {
3100        //!     a.or(b)
3101        //! }
3102        //! 
3103        //! let present_1 = Some(10);
3104        //! let present_2 = Some(20);
3105        //! let empty = None;
3106        //! assert_eq!(pick_number(present_1, present_2), present_1);
3107        //! assert_eq!(pick_number(present_1, empty), present_1);
3108        //! assert_eq!(pick_number(empty, present_1), present_1);
3109        //! assert_eq!(pick_number(empty, empty), empty);
3110        //! ```
3111        //! 
3112        //! ## `Parser` trait and `construct!` macro
3113        //! 
3114        //! [`Parser`] trait defines a context for values and gives access to `Functor` laws and [`construct!`]
3115        //! macro allows to compose several values according to `Applicative` and `Alternative` laws.
3116        //! 
3117        //! ## So why use `Applicative Functors` then?
3118        //! 
3119        //! As a user I want to be able to express requirements using full power of Rust algebraic
3120        //! datatypes: `struct` for product types and `enum` for sum types. To give an example -
3121        //! `cargo-show-asm` asks user to specify what to output - Intel or AT&T asm, LLVM or Rust's MIR
3122        //! and opts to represent it as one of four flags: `--intel`, `--att`, `--llvm` and `--mir`. While
3123        //! each flag can be though of a boolean value - present/absent - consuming it as an `enum` with four
3124        //! possible values is much more convenient compared to a struct-like thing that can have any
3125        //! combination of the flags inside:
3126        //! 
3127        //! ```no_check
3128        //! /// Format selection as enum - program needs to deal with just one format
3129        //! enum Format {
3130        //!     Intel,
3131        //!     Att,
3132        //!     Llvm,
3133        //!     Mir
3134        //! }
3135        //! 
3136        //! /// Format selection as struct - can represent any possible combination of formats
3137        //! struct Formats {
3138        //!     intel: bool,
3139        //!     att: bool,
3140        //!     llvm: bool,
3141        //!     mir: bool,
3142        //! }
3143        //! ```
3144        //! 
3145        //! `Applicative` interface gives just enough power to compose simple parsers as an arbitrary tree
3146        //! ready for consumption.
3147        //! 
3148        //! As a library author I need to be able to extract information from the tree constructed by user
3149        //! to generate `--help` information and do command line completion. As long as the tree uses only
3150        //! `Applicative` powers - it is possible to evaluate it without giving it any input.
3151        //! Adding `Monadic` powers (deciding what to parse next depending on the previous input) would
3152        //! make this impossible.
3153        //! 
3154        //! So `Applicative Functors` sits right in the middle between what users want to express and
3155        //! library can consume.
3156        //! 
3157        //! To recap - all sorts of Functors listed here only define laws to how individual parts are
3158        //! composed, how values in context can be transformed and how pure values can be turned into a
3159        //! functor, but not how the values are parsed or how they can be extracted.
3160        //! 
3161        //! ## Putting the values into a context
3162        //! 
3163        //! Similarly to how `Reader` defined above `bpaf`'s `Parsers` don't actually have values inside
3164        //! until they are executed. Instead starting points ([`flag`](NamedArg::flag), [`positional`],
3165        //! [`argument`](NamedArg::argument), etc) define what exactly needs to be consumed, various mapping
3166        //! functions define transformations, [`construct!`] composes them and defines the relative order
3167        //! values should be consumed. Not everything present inside [`Parser`] can be repesented in terms
3168        //! of plain applicative functors - specifically [`parse`](Parser::parse) is not and it is best
3169        //! though of as a function that takes one applicative and gives a different applicative back.
3170        //! The actual values will show up inside once `bpaf` starts running the [`OptionParser`] with
3171        //! [`run`](OptionParser::run).
3172        //! 
3173        //! ## Taking the results out
3174        //! 
3175        //! The rest of the execution is relatively simple: getting console arguments from OS, doing the
3176        //! initial split into short/long flags and standalone words, disambiguating groups of short
3177        //! options from short options with attached values and applying all the transformations like
3178        //! `Reader::run` above would do.
